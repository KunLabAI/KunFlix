# 后端积分模型审查与商业评估报告

## 1. 设计合理性审查

### 1.1 核心流程
- **生成与消耗**: 目前通过 `calculate_credit_cost` 实现并在 `chats.py` / `orchestrator.py` 中触发。逻辑本身合理，但缺乏预先验证。
- **过期机制**: **缺失**。没有积分过期机制（例如，月度订阅积分通常不应结转到下个月）。
- **冻结机制**: **缺失**。`User` 或 `Admin` 表中没有冻结账户或积分的状态字段。
- **退款机制**: **缺失**。目前仅存在手动的“管理员调整 (admin_adjust)”。对于部分消耗 token 但最终失败的生成任务，缺乏自动退款逻辑（虽然目前部分代码仅在成功时调用计费，但流式传输失败可能需要部分计费或全额回滚）。
- **货币兑换**: 前端假设 `1 积分 = $0.01 美元`。后端将积分处理为原始浮点数。

### 1.2 数据完整性与安全性
- **精度问题**: `credits` 和 `amount` 使用 `sa.Float` 类型。**风险**: 浮点数误差可能会累积。
  - *建议*: 迁移到 `DECIMAL(18, 4)` 或 `Integer` (微积分单位) 以确保财务准确性。
- **并发问题 (竞态条件)**:
  - 当前逻辑:
    ```python
    balance_before = entity.credits or 0
    entity.credits = max(0, balance_before - credit_cost)
    ```
  - **严重风险**: 高并发场景（例如，同一用户的多个智能体并行运行）将导致**更新丢失**。如果两个进程读取到相同的 `balance_before`，后一个写入将覆盖前一个扣除操作。
  - *修复*: 使用原子数据库更新: `UPDATE users SET credits = credits - :cost WHERE id = :id`。
- **负余额风险**:
  - `max(0, ...)` 虽然防止了数据库字段变为负数，但允许**服务滥用**。只有 0.001 积分的用户可以触发昂贵的任务，最终余额变为 0 但获得了完整的服务价值。
  - *修复*: 在执行*前*实现 `check_balance_sufficient()` 检查。
- **幂等性**:
  - 虽然记录了 `session_id`，但单个消息/任务执行 ID 在 `credit_transactions` 中没有唯一约束。重试可能会导致重复计费。

### 1.3 数据库结构
- **表结构**: `credit_transactions` 结构良好，包含 `balance_before`, `balance_after`, 和 `metadata_json`。
- **索引**: `user_id` 已建立索引。`created_at` 也应建立索引以用于报表查询。

## 3. 积分模型优化建议 (Action Plan)

基于上述 2026 年最新成本数据，建议从以下三个维度重构积分模型：

### 3.1 核心汇率锚定
建议维持 **1 积分 = $0.01 USD** 的锚定汇率。
*   **理由**:
    *   **直观**: 用户易于理解 (100 积分 = $1)。
    *   **精度**: 当前最低成本模型 (GPT-5 Nano/DeepSeek Cache) 成本约 $0.03/1M，即 3 积分/1M，仍在整数积分表达范围内，无需过度分割。
    *   **兼容**: 现有数据库 `Float` 类型可以平滑过渡，仅需前端展示调整。

### 3.2 动态定价策略 (Markup Strategy)
建议采用 **"基础成本 + 运营缓冲"** 的动态加成模型，而非固定倍率。

| 模型层级 | 代表模型 | 成本 (积分/1M Input) | 建议定价 (积分/1M Input) | 加成倍率 | 策略逻辑 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **入门/引流** | DeepSeek V3, GPT-5 Nano | 5 ~ 28 | **30** | ~1.1x - 6x | 低价引流，通过高频小任务培养习惯。利用缓存命中覆盖成本。 |
| **主力/通用** | Gemini 3.1 Flash, MiniMax | 25 ~ 30 | **50** | ~1.6x - 2x | 平衡性能与利润，作为默认推荐模型。 |
| **进阶/增强** | GPT-5 Mini, Haiku 4.5 | 25 ~ 100 | **120** | ~1.2x - 4x | 针对有一定质量要求的中间场景。 |
| **旗舰/专业** | GPT-5.2, Sonnet 4.6 | 175 ~ 300 | **400** | ~1.3x - 2.2x | 高价值任务，确保单次调用利润绝对值。 |
| **顶配/推理** | Opus 4.6, o1 | 500 ~ 1500 | **2000** | ~1.3x - 4x | 限制级使用，高定价筛选高价值需求，防止滥用。 |

从“按张计费”全面转向 **"按分辨率/时长阶梯计费"**。

## 4. 完整性分析


### 3.1 缺口
- **定价策略**:
  - 当前: `input_credit_per_1m`, `output_credit_per_1m`.
  - 缺失: **阶梯定价** (例如，10亿 tokens 后更便宜), **缓存输入** 定价 (许多提供商对缓存上下文提供 50-90% 的折扣)。
- **搜索计费**:
  - `search` 的 scale=1 (按次查询)。确保这能覆盖实际 API 成本 (例如，Google Search API ~$5/1000 次查询 = $0.005/次 = 0.5 积分)。
- **扩展功能**:
  - 不支持“每日免费积分”（常见的留存策略）。
  - 计费逻辑中没有区分“订阅”与“按量付费”（前端/Stripe 处理了，但后端积分池是统一的）。

### 3.2 风险
- **汇率锁定**: 如果美元/人民币汇率波动显著，固定的积分价格可能会导致无利可图或对本地用户过于昂贵。
- **合规性**: 后端没有税务计算或发票生成逻辑。

## 4. 交付与验收标准

### 4.1 风险矩阵

| 风险 | 严重程度 | 缓解措施 |
| :--- | :--- | :--- |
| **竞态条件 (积分丢失)** | **P0** | 实现原子 SQL 更新或行级锁。 |
| **负余额滥用** | **P1** | 增加运行前余额检查。 |
| **浮点精度误差** | **P2** | 将列迁移至 DECIMAL/Integer。 |
| **缺失过期机制** | P3 | 在积分批次中增加 `expires_at` (需要变更 Schema)。 |

### 4.2 建议优化方案

**1. 原子扣除 SQL (SQLAlchemy)**
代替 Python 中的“读-改-写”模式：
```python
# 错误示范 (BAD)
user.credits = user.credits - cost

# 正确示范 (GOOD)
stmt = (
    update(User)
    .where(User.id == user_id)
    .where(User.credits >= cost)
    .values(credits=User.credits - cost)
    .execution_options(synchronize_session="fetch")
)
result = await session.execute(stmt)
if result.rowcount == 0:
    raise InsufficientCreditsError()
```

**2. 运行前检查 (Check-Before-Run)**
添加中间件或装饰器来检查余额预估：
```python
estimated_cost = estimate_tokens(input_text) * rate * 1.2  # 20% 缓冲
if user.credits < estimated_cost:
    raise PaymentRequired("Insufficient credits")
```

**3. 验收测试计划**
- **并发测试**: 10 个并发请求，每次消耗 10 积分，针对 100 积分余额。结果必须是余额准确为 0，且产生 10 条交易记录。
- **零余额测试**: 针对 0 余额请求消耗 1 积分。必须在 LLM 调用*前*失败。
- **精度测试**: 增加 0.0001 积分 10,000 次。结果必须准确为 1.0。
